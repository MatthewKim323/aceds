# ACE — One-Pager

**Aggregate grade prediction + IP schedule optimization, trained on 17 years of UCSB grade data.**

---

## The problem

Every UCSB registration cycle, 25,000 undergraduates pick sections from a catalog of ~4,400 offerings, largely on the basis of three signals: course title, meeting time, and word-of-mouth. There is no sanctioned way to ask _"Of the three Econ 10A sections, which one historically gives an average grade closest to a B+ with a professor who's rated above 4.0 and doesn't start at 8am?"_ — despite every input to that question being public data.

Students answer it anyway, poorly, through GauchoSpace threads and RateMyProfessors searches. The top-rated professor frequently has a mean section GPA two standard deviations above the rest of the department — a feature of the current information asymmetry, not a bug.

## The tool

ACE ingests the Daily Nexus grade dump (2009 Fall → 2026 Winter, 104,549 rows), joins it to 6,028 RateMyProfessor profiles and the live UCSB catalog, and serves two things:

1. **`POST /predict`** — an XGBoost regressor that returns the expected mean GPA and prediction standard deviation for a given section, keyed by `(course, instructor, quarter)`. Test RMSE **0.234**, calibration slope **1.03** on a strictly-held-out 2026 Winter test set.
2. **`POST /optimize`** — a PuLP/CBC integer program that picks a non-conflicting schedule maximizing a weighted combination of predicted grade, professor rating, time convenience, and seat availability. Solves 3–6 course problems with 6–12 candidates each in **p50 44 ms / p95 83 ms** across 240 synthetic benchmarks.

A React dashboard on top: `/explorer` for catalog search, `/schedule` for optimizer results, `/gradpath` for major-requirement planning, `/dashboard` for personal GPA + unit tracking. Demo mode is 50 synthetic students — judges click "try demo" and land in a populated account.

## Why the numbers are real

- **Temporal splits, no leakage.** Every historical aggregate is computed expanding from the training-side of the time cutoff. Val / test quarters the model has never seen.
- **Missingness preserved, never imputed.** Imputation of `rmp_rating` with department means was the obvious win and would have made the model look better than it is. We resisted.
- **Ablation-driven.** History does the work. Removing RMP costs 0.002 RMSE. Removing the historical aggregates costs 0.059 — a 25% regression. This is the "we're not RMP for grades" slide.
- **Cold-start is explicit.** New instructor + new course + no pair history → the returned `predicted_gpa_std` widens, and the optimizer's preference scoring discounts accordingly.

## Stack

- **Data**: Python, pandas, httpx, a fuzzy-matching RMP scraper with a confidence field.
- **Model**: XGBoost + ElasticNet + a hard-cascade heuristic baseline. All three reproducible from `data_pipeline/scripts/`.
- **Optimizer**: PuLP + CBC integer program.
- **Backend**: FastAPI, pydantic, Supabase (Postgres + Auth), structlog, deployed to Fly.io.
- **Frontend**: React 19, Vite, Motion. No component library — a handwritten design system with CSS variables.
- **CI**: GitHub Actions — ruff + mypy + pytest + Docker smoke test + frontend typecheck & build on every PR.

## What's next

- Per-student grade prediction (currently aggregate-only). The synthetic-student layer is a placeholder; a real per-student model needs opted-in historical grades. Privacy-first — never leaves the user's session without explicit consent.
- Full major-requirement semantic search (course embeddings already shipped locally via `sentence-transformers/all-MiniLM-L6-v2` — `15_embeddings.py`, 1,309 × 384d, no API key). Next: wire into the Explorer's "similar courses" and the optimizer's redundancy penalty.
- Live Supabase deploy for a hosted judge demo. Every dependency is in place; it's a `fly deploy` + a Netlify push away.
